# 0.21 doc-sweep — issues report

Captures the friction I hit while writing the three new monitor
examples + the root-gated subscribe test, plus the doc
mis-statements that surfaced as a result. Resolved in commit
`761b6d4`. Filed so you can decide whether any of these deserve
follow-up work beyond what already shipped.

## TL;DR

One genuine API discovery (`Monitor::run_for` returns a `!Send`
future), three doc/example mistakes I made on the first pass,
two clippy nits, one symlink gotcha. The `!Send` future story is
the only one with material impact — it propagated through
README + ASYNC_GUIDE + ARCHITECTURE + CLAUDE.md before I caught
it.

## 1. `Monitor::run_for(...)` future is `!Send`

**Severity:** high — the docs were actively wrong until I fixed
them in this commit.

**What I discovered:** `tests/monitor_send.rs` asserts
`Monitor: Send`, `MonitorBuilder: Send`, `ShardedRunner: Send`,
`EventStream<M>: Send` — all four pass. The 0.21 H.2 narrative
("Send sweep") and the audit summary both treat this as the full
story. It isn't. The *future* returned by
`Monitor::run_for(d).await` is `!Send` because the run loop
holds an `AsyncCapture<S>` across awaits, and `AsyncCapture`
borrows the `!Sync` mmap ring. `tokio::spawn(monitor.run_for(d))`
fails to compile with:

```
error[E0277]: `*const ()` cannot be sent between threads safely
note: future is not `Send` as it awaits `AsyncCapture` operations
```

**How I hit it:** I wrote `tests/monitor_lo_subscribe.rs` with
`tokio::spawn(async move { monitor.run_for(...).await })` and
then pulled the subscriber stream from the test task. Refused
to compile. Rewrote to use `#[tokio::test(flavor =
"current_thread")]` + `tokio::select!` (same pattern as the
existing `tests/monitor_lo_dispatch.rs`).

**Doc fallout (corrected in this commit):**

| Doc | Original claim | Fixed claim |
|---|---|---|
| README.md | "Monitor is Send — no `current_thread` ceremony" | "Monitor value is Send; `run_for` future stays !Send because AsyncCapture borrows the mmap ring across awaits. Use `tokio::select!` or ship via `ChannelSink`." |
| docs/ASYNC_GUIDE.md | "Monitor lives on the default multi-thread runtime" | Same Send-future caveat spelled out |
| docs/ARCHITECTURE.md | "Users can `tokio::spawn` the future and pull subscribers from another task" | Reversed — explicitly says you cannot spawn the run-loop future |
| netring/CLAUDE.md | "Monitor now lives on the default multi-thread runtime" | Same caveat |

**Why this matters past the doc fix:** the audit-driven sweep
in commit `3741de4` titled itself "Send Monitor" and the
0.21 CHANGELOG entry headlines "Send Monitor + sharding + …".
A reader who sees that headline expects the run loop to be
spawnable — that's the typical reason a type needs to be `Send`.
It isn't here, and the constraint is structural (mmap borrows),
not something a future Cargo feature could relax. Worth
considering whether 0.22 should either:
- Type-erase the run loop behind a `Send` boundary (likely
  requires copying owned packets instead of zero-copy borrows
  on the run path — perf trade-off), or
- Explicitly rename the headline so users don't assume more
  than the assertion delivers.

The honest one-line summary is "Monitor value is Send so it
composes with multi-thread runtimes; the run-loop future is
still !Send by construction" — which is what the docs now say.

## 2. API mistakes I made on the first example pass

These are pure user errors but worth recording because they
suggest where the API surface has friction points that better
prelude exports or doctest examples could smooth out.

### 2a. `MinSeverity::info()` doesn't exist

I wrote `.layer(MinSeverity::info())` in `eve_to_filebeat.rs`.
The actual API is `MinSeverity::at_least(Severity::Info)`, with
convenience constructors only for `warning()` and `error()`.

Asymmetry is deliberate (Info-and-up is a no-op for the common
case so the constructor was left off), but `MinSeverity::info()`
is the obvious-looking call and it compiles in my head from the
`Severity::Info` variant. Worth either:
- Adding `MinSeverity::info()` as a `pub const` alias of
  `at_least(Severity::Info)`, or
- Documenting the omission inline so the next person doesn't
  guess.

### 2b. `MonitorBuilder::run_for` doesn't exist

I wrote `Monitor::builder().interface(...)....run_for(d)`,
forgetting the `.build()?` in between. The error message was
clear (`no method named run_for found for MonitorBuilder`), but
this is the second time I've made this mistake in this codebase
— the builder/handle split for `run_*` methods is unusual
relative to e.g. `AsyncCapture::open(...)` which is one call.

Not actionable as an API change (forcing `.build()` makes the
validation errors explicit), but worth a sentence in
`docs/API_OVERVIEW.md`'s builder table making clear that
`run_*` lives on `Monitor`, not `MonitorBuilder`. The new table
I added in this commit shows this implicitly by putting
`run_*` under "Run modes" rather than "Builder surface" — but
a one-line cross-reference would help.

### 2c. Top-level `README.md` is a symlink to `netring/README.md`

First Edit attempted on `/var/home/mpardo/git/netring/README.md`
got rejected ("refusing to write through symlink"). Had to read
the symlink target with `readlink` and edit the inner file.

This is a sandbox harness quirk, not a netring issue, but worth
flagging: the same edit had to be retried twice (once for each
path that the tool tracker considered separate). If you ever
want to change docs from a Claude session, the inner path
(`netring/README.md`) is the one to target.

## 3. Lint nits cargo fmt / clippy surfaced

Both caught after I'd written the examples; both auto-fixable;
neither pointed to real bugs.

### 3a. `clippy::doc_lazy_continuation` on README.md

My migration link paragraph started lines with `+ broadcast subscribers`
which the clippy 1.95 doc lint interprets as continuation of a
phantom list item. Fixed by replacing the `+`-separated phrase
with comma-separated.

This is one of clippy 1.95's stricter doc lints and likely to
trip again as the README evolves — the build matrix probably
needs to surface the rendered doc page once or twice per cycle
to spot these before they break CI.

### 3b. `clippy::needless_borrows_for_generic_args`

`.with("src", &format!("{}", evt.key.a))` — clippy wants the `&`
dropped since the `AnomalyWriter::with` argument generic accepts
owned strings. Fixed in both new examples.

The existing `monitor/basic.rs` writes `.with("sni",
hs.sni.as_deref().unwrap_or("<none>"))` (no `&`), so the
codebase convention is the bare argument. Good catch by clippy
but pre-existing examples already followed the rule, so no
codebase change needed.

## 4. `cargo fmt` touched `benches/anomaly.rs`

Not a file I changed. `cargo fmt` reformatted the
`#![allow(deprecated)]` line — turned a one-line attribute +
comment into two lines. Committed alongside the rest.

Worth knowing about: a fresh `cargo fmt` run can mutate files
you didn't intend to touch, which complicates "small focused
commit" hygiene. Not a netring issue.

## 5. What I'd recommend (small)

In rough priority order:

1. **Reframe the "Monitor is Send" story.** The CHANGELOG, the
   0.21 H.2 narrative, and the prior README all overpromise.
   This commit corrects the docs but the 0.21.0 CHANGELOG entry
   header still reads "Send Monitor + sharding + …". Consider
   either renaming on tag day to "Send Monitor value + …" or
   adding a one-paragraph "what Send buys you here" note in
   the CHANGELOG entry body.

2. **Add `MinSeverity::info()`.** One-line const constructor.
   Removes a foot-gun that will hit other first-time users.

3. **Cross-reference `Monitor` vs `MonitorBuilder` method
   placement** in API_OVERVIEW. The new table I added gets
   close but a one-sentence "run modes live on `Monitor`, not
   the builder — call `.build()?` first" note above the table
   would close it.

4. **Document the `!Send` run-loop future constraint in the
   migration guide.** Right now `docs/MIGRATING_0.20_TO_0.21.md`
   says "drop `flavor = "current_thread"`" without the
   asterisk. If you keep that header, the asterisk needs to
   land in the body.

5. **Watch clippy's doc lints in CI.** `doc_lazy_continuation`
   on README markdown is the kind of thing that's easy to
   reintroduce.

None of these block the 0.21.0 tag.
