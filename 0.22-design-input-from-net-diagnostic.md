# 0.22 design input — what `net_diagnostic` taught us

A design exercise grounded in
[`netring/examples/monitor/net_diagnostic.rs`](netring/examples/monitor/net_diagnostic.rs)
(306 LoC, committed as `4d2baad`). The example composes three
operationally-common signals — ICMP Destination Unreachable
alerts, TCP connection RESETs, and bandwidth-by-application —
in one `Monitor::builder()` pipeline. Writing it surfaced
ergonomic gaps that compound across realistic deployments.

Backward compatibility is on the table. Both crates are
single-maintainer; we can ship breaking changes freely.

## TL;DR — the top five

1. **Bandwidth-by-app is a builtin, not a user concern.** Every
   real deployment wants it; hand-rolling ~80 LoC for the
   five-line use case is wrong. Ship
   `MonitorBuilder::bandwidth_by_app()` + a `BandwidthSnapshot`
   tick event.
2. **The Protocol/Event matrix is asymmetric and forces
   duplicate handlers.** `FlowPacket<Tcp>` + `FlowPacket<Udp>`
   handlers are byte-identical when they only need `key`/`len`.
   Add `AnyFlowPacket` (and `AnyFlowEnded` / `AnyFlowStarted`)
   that fires for any L4.
3. **ICMP is a half-shipped feature.** flowscope ships
   `IcmpType::error_inner()` returning the embedded 5-tuple,
   but nothing in netring correlates it back to a known flow.
   The "did this ICMP kill an active flow?" join requires hand
   wiring. Ship `IcmpCorrelated` event + classified
   `IcmpErrorKind` enum.
4. **Ctx state ergonomics scale poorly past two slots.** Three
   state slots already forced an awkward "read by value, borrow
   last" dance in the tick handler. Ship `ctx.snapshot::<R>()`
   over a derive macro, or move tick reports to a
   `ReportSink` model entirely.
5. **The prelude is incomplete and the public types live in
   inconsistent places.** `FlowPacket<P>` isn't reexported.
   ICMP code enums require either `flowscope::icmp::*` or a
   private path. Sharpen both crates' surfaces.

The rest of this document expands each, plus several smaller
items, plus a sketch of what a more aggressive redesign could
look like.

## 1. Pain points (concrete)

Quotes are from `net_diagnostic.rs`. Line numbers reference the
shipped file.

### 1.1 `FlowPacket<P>` isn't in the prelude (L57–58)

```rust
use netring::prelude::*;
use netring::protocol::event_typed::FlowPacket;
```

The prelude exports `FlowStarted`, `FlowEnded`, `FlowEstablished`,
`AnyFlowAnomaly`, `Tick` — but `FlowPacket`, `FlowTick`,
`ParserClosed` are missing. Compile error → import hunt → fix.
Pure inconsistency tax; no design tension.

### 1.2 ICMP code enums hide behind a private path (L54–56)

```rust
use flowscope::icmp::{
    IcmpMessage, IcmpType, Icmpv4DestUnreachCode, Icmpv4Type, Icmpv6DestUnreachCode, Icmpv6Type,
};
```

First attempt: `use flowscope::icmp::types::Icmpv6DestUnreachCode`
— `types` is private. Rescued by `pub use types::*` in
`flowscope::icmp::mod`. The doc-link bait
(`Icmpv6DestUnreachCode` lives at `flowscope::icmp::types::…` in
the rustdoc) sends users to a dead address.

### 1.3 ICMP `DestinationUnreachable` codes don't share a
canonical shape (L108–140)

```rust
fn classify_unreachable(msg: &IcmpMessage) -> Option<(&'static str, Severity)> {
    match &msg.ty {
        IcmpType::V4(Icmpv4Type::DestinationUnreachable { code, .. }) => match code {
            Icmpv4DestUnreachCode::Host | Icmpv4DestUnreachCode::DestHostUnknown => …,
            Icmpv4DestUnreachCode::Port => …,
            …
        },
        IcmpType::V6(Icmpv6Type::DestinationUnreachable { code, .. }) => match code {
            Icmpv6DestUnreachCode::AddressUnreachable => …,
            Icmpv6DestUnreachCode::PortUnreachable => …,
            …
        },
        _ => None,
    }
}
```

30 lines of nested matching, with code-by-code mapping written
twice (v4 then v6) for what is really one logical concept:
"this destination wasn't reachable, here's why". Every consumer
will write a near-identical function.

flowscope already has `IcmpType::short_kind()` (returns
`"dest_unreachable"`, `"time_exceeded"`, etc.) but stops at the
type — not at the code's operational meaning. We need one more
level of shared classification.

### 1.4 `IcmpType` is `#[non_exhaustive]` despite having only
V4/V6 variants

```rust
fn icmp_family_str(msg: &IcmpMessage) -> &'static str {
    match msg.ty {
        IcmpType::V4(_) => "v4",
        IcmpType::V6(_) => "v6",
        _ => "other",   // unreachable, but the compiler insists
    }
}
```

The `_` arm is dead code today. Either the third variant
exists in flowscope's roadmap (so the `#[non_exhaustive]` is
forward-looking — fine, but document the planned variant) or
the marker is over-cautious and should come off.

### 1.5 Per-L4 duplication for protocol-agnostic state (L264–272)

```rust
.on_ctx::<FlowPacket<Tcp>>(|evt: &FlowPacket<Tcp>, ctx: &mut Ctx<'_>| {
    let stats = ctx.state_mut::<AppBandwidth>();
    bandwidth_bucket(stats, &evt.key, true).add(evt.len);
    Ok(())
})
.on_ctx::<FlowPacket<Udp>>(|evt: &FlowPacket<Udp>, ctx: &mut Ctx<'_>| {
    let stats = ctx.state_mut::<AppBandwidth>();
    bandwidth_bucket(stats, &evt.key, false).add(evt.len);
    Ok(())
})
```

Two handlers, byte-identical except for the type parameter and
a `bool`. The type parameter only matters because the protocol
classifier is `is_tcp: bool` — and the only reason that exists
is to pick the right "other/tcp" vs "other/udp" bucket.

The protocol is already in `key.proto` (an `L4Proto` enum). The
parametric dispatch buys us no type safety here; both handlers
just need "a flow packet, any L4".

### 1.6 TCP RST filtering happens in the handler (L226–229)

```rust
.on_ctx::<FlowEnded<Tcp>>(|evt: &FlowEnded<Tcp>, ctx: &mut Ctx<'_>| {
    if evt.reason != EndReason::Rst {
        return Ok(());
    }
    …
```

Every "RST alert" handler will repeat this filter. There's no
typed event for "TCP flow ended via RST" — the user does the
narrowing themselves. Same shape would apply to
"FlowEnded::ParserError" or "FlowEnded::IdleTimeout" cases.

### 1.7 Three state slots → awkward tick handler (L283–293)

```rust
.tick(Duration::from_secs(5), |_tick: &Tick, ctx: &mut Ctx<'_>| {
    let resets = *ctx.state_mut::<TcpResetCount>();          // Copy
    let icmp = *ctx.state_mut::<IcmpUnreachableCount>();    // Copy
    let stats = ctx.state_mut::<AppBandwidth>();            // &mut, last
    print_bandwidth_report(stats, &resets, &icmp);
    Ok(())
})
```

This compiles because each `state_mut::<T>()` is a fresh borrow
that ends at `;`. But it required deriving `Copy` on
`TcpResetCount` and `IcmpUnreachableCount` specifically to
support this read-by-value step.

For a 4th state slot, you start running out of patterns.
`split_state_*` helpers in `Ctx` go up to 4-way
(state + sink + counter + state-sink-counter) but no further,
and they're for **simultaneous** access not the tick-report
pattern. A 5-way report breaks.

### 1.8 Bandwidth-by-app classifier needs a fallback by L4 (L142–157)

```rust
fn bandwidth_bucket<'a>(
    stats: &'a mut AppBandwidth,
    key: &FlowKey,
    is_tcp: bool,
) -> &'a mut AppBucket {
    if let Some(label) = key.protocol_label() {
        stats.by_app.entry(label).or_default()
    } else if is_tcp {
        &mut stats.other_tcp
    } else {
        &mut stats.other_udp
    }
}
```

`protocol_label()` is `Option<&'static str>`. When it returns
`None` (random ephemeral port pair), we fall back to a generic
"other/tcp" or "other/udp" bucket. The L4-conditional fallback
is the only reason this function needed a `bool` parameter
(which then forced the handler split above).

A `key.app_or_l4_label()` returning `&'static str` ("http" /
"https" / "tcp" / "udp" / "icmp") would skip the entire
problem.

### 1.9 `FlowStats` direction summation by hand (L233)

```rust
let total_bytes = evt.stats.bytes_initiator + evt.stats.bytes_responder;
```

flowscope has `FlowStats::bytes_total()`. I didn't notice and
hand-summed. Discoverability gap. Same applies to
`pkts_total()`, `mean_pkt_size()`, etc. — neither obviously
exists.

### 1.10 The tick payload (`&Tick`) is rarely used

```rust
.tick(Duration::from_secs(5), |_tick: &Tick, ctx: &mut Ctx<'_>| {
```

In four shipped tick handlers across the example suite, none
of them read the `Tick` payload. It's a parameter you destructure
into `_` to keep the type checker happy. If we kept the model,
fine — but if we redesign, `Tick`'s payload should either earn
its keep or be inferred away.

### 1.11 Three protocols → three explicit registrations (L208–211)

```rust
.protocol::<Tcp>()
.protocol::<Udp>()
.protocol::<Icmp>()
```

For "watch this NIC for everything" the user enumerates every
marker. There's no `.all_l4()`. Tcp/Udp/Icmp is the natural
unit; people who want one of them already do `.protocol::<Tcp>()`.

### 1.12 ICMP is conceptually misplaced as a "Protocol"

`on_ctx::<Icmp>` receives `&IcmpMessage` directly, not a
`FlowStarted<Icmp>` / `FlowEnded<Icmp>` lifecycle pair. ICMP
isn't flow-shaped — it's request/reply at best (Echo/Reply
pairs), event-shaped at worst (Destination Unreachable
broadcasts). But the registration ceremony reads identically
to a real flow-tracked protocol, which mislead me at first
into expecting lifecycle events for ICMP.

## 2. Themes

Six recurring patterns across §1:

**T1. The Protocol marker buys us less than the ceremony suggests.**
For lifecycle events on Tcp/Udp it's genuine ("this TCP flow
started"). For per-packet bandwidth tracking it's noise. For
ICMP it's actively misleading. We should split the marker into
"flow-tracked protocol" (Tcp/Udp/+L7) and "message-only
listener" (Icmp).

**T2. There's no first-class "report" abstraction.**
Periodic snapshots are written via `.tick(period, ...)`, and
each handler hand-rolls the data plumbing. EveSink ships
structured alerts. Nothing ships structured reports. We don't
need to reinvent Prometheus, but we do need a primitive that
catches the "every N seconds, snapshot these N counters into a
typed struct and ship it" pattern.

**T3. The N-state-slot tick handler is fragile.**
The Ctx model assumes handlers want **one or two** slots open.
Reports want **all** of them, sequentially. Either the model
grows to support that ergonomically or reports leave the Ctx
model entirely.

**T4. Bandwidth tracking is bandwidth tracking.**
Every operations dashboard does it the same way: bytes-per-app,
per-host, per-flow, sometimes per-direction. Shipping it as a
first-class primitive collapses 80 LoC of handler/state/tick to
two builder lines.

**T5. Discoverability matters more than completeness.**
Flowscope already has `FlowStats::bytes_total()`,
`IcmpType::error_inner()`, `IcmpType::short_kind()`,
`FiveTupleKey::protocol_label()`. Half of them I found while
writing the example, half I found while writing this report.
The crate's surface is rich but the prelude leaves users
hunting.

**T6. Backward-compat-free is permission, not direction.**
The 0.21 cycle already broke things twice (key narrowing, Send
sweep). Each break shipped a recipe in
`MIGRATING_0.20_TO_0.21.md`. The volume of small breaking
changes the items below propose is comparable to one Phase. We
can bundle them into a single 0.22 cycle without burdening
users beyond what 0.20→0.21 already did.

## 3. Small flowscope additions

These are pure-additive (or near-additive) flowscope-side
changes that simplify the netring story.

### F1. `DestUnreachableKind` enum + accessor

```rust
// flowscope::icmp::DestUnreachableKind
#[non_exhaustive]
pub enum DestUnreachableKind {
    Host,
    Port,
    Network,
    Protocol,
    AdministrativelyProhibited,
    FragmentationNeeded,
    Other,
}

impl IcmpType {
    pub fn dest_unreachable_kind(&self) -> Option<DestUnreachableKind>;
}
```

Maps the v4 + v6 code enums onto one operational vocabulary.
`net_diagnostic`'s 30-line `classify_unreachable` function
becomes a one-line match.

### F2. `IcmpType::is_error()` is already shipped — surface it

It exists (line 294 of `flowscope/src/icmp/types.rs`) but
isn't mentioned in any netring example or doc. Add to the
flowscope prelude alongside `error_inner()` and document a
small "icmp triage" recipe.

### F3. `FiveTupleKey::app_label()` returning `&'static str`

A non-`Option` companion to `protocol_label()`. Falls back to
the canonical L4 name (`"tcp"` / `"udp"` / `"icmp"`) when
neither port maps. Closes the gap that forced the `is_tcp:
bool` parameter in `bandwidth_bucket`.

```rust
impl FiveTupleKey {
    /// Always-some companion to [`Self::protocol_label`].
    /// Returns the L4-canonical fallback (`"tcp"`, `"udp"`,
    /// `"icmp"`) when no well-known port matches.
    pub fn app_label(&self) -> &'static str;
}
```

### F4. `FlowStats` summary helpers — review what's missing

`bytes_total()` exists. Add (or surface) `pkts_total()`,
`mean_pkt_size()`, `bytes_per_side(FlowSide)`, optionally
`duration()` (needs `start_ts`). Make discoverability
explicit by linking them from the `FlowStats` struct doc.

### F5. `IcmpCorrelated` event scaffolding

flowscope already gives you `IcmpType::error_inner() ->
Option<(&'static str, &IcmpInner)>`. What's missing is the
**lookup**: given an `IcmpInner` carrying (src_ip, dst_ip,
proto, src_port, dst_port), is there a live flow that matches?

Option A: flowscope ships `FlowTracker::lookup_inner(inner) ->
Option<FlowKey>` (synchronous, accepts the partial 5-tuple,
returns canonical key if a flow matches).

Option B: netring handles the join (less ideal — duplicates
flowscope's internal index).

This is the highest-leverage addition: it lights up the
"connection died because of an ICMP that explained why" join
that every L4 monitor wants.

### F6. Public re-export hygiene

The `types` module-private-with-`pub use *` pattern in
`flowscope::icmp` breaks rustdoc links. Either:
- Move types to `flowscope::icmp::types::*` and make `types`
  public (no `pub use *` shim), or
- Document the canonical path is `flowscope::icmp::*` and
  delete the `types::` references from generated docs.

### F7. `protocol_label()` extensibility

Today the well-known table is hard-coded. Site deployments
have custom services on non-standard ports ("our internal
gRPC service on 8765"). Either:

```rust
pub fn protocol_label_with(
    proto: L4Proto,
    src_port: u16,
    dst_port: u16,
    custom: &impl Fn(L4Proto, u16) -> Option<&'static str>,
) -> Option<&'static str>;
```

or — better — `FlowTracker::with_label_table(table)` so the
labels propagate through `FiveTupleKey::protocol_label()`
calls downstream.

### F8. `FlowSide`-aware byte accumulation as a primitive

A `RollingRate<K, V>` correlate primitive (parallel to
`TimeBucketedCounter<K>`) that buckets `(K, V)` over a
sliding window. `bandwidth_by_app` is a special case:
`K = &'static str` (app label), `V = u64` (bytes/second
rolling).

```rust
// flowscope::correlate::RollingRate
pub struct RollingRate<K, V> { ... }
impl<K, V> RollingRate<K, V> {
    pub fn record(&mut self, k: K, v: V, ts: Timestamp);
    pub fn rate(&self, k: &K, now: Timestamp) -> V; // per-second
    pub fn snapshot(&self, now: Timestamp) -> impl Iterator<Item = (&K, V)>;
}
```

## 4. Small netring additions

### N1. Prelude completeness

```rust
// netring::prelude
pub use crate::protocol::event_typed::{
    AnyFlowAnomaly, FlowEnded, FlowEstablished, FlowPacket, FlowStarted, FlowTick,
    ParserClosed, Tick,
};
pub use flowscope::icmp::{
    DestUnreachableKind, IcmpMessage, IcmpType,
};
```

One commit.

### N2. `AnyFlowPacket` / `AnyFlowEnded` event

A protocol-agnostic event that fires for any L4 protocol. The
payload carries `proto: L4Proto` so the handler can branch if
needed.

```rust
pub struct AnyFlowPacket {
    pub key: FlowKey,
    pub proto: L4Proto,
    pub side: FlowSide,
    pub len: usize,
    pub tcp: Option<TcpInfo>,
    pub ts: Timestamp,
}
```

Collapses the two `FlowPacket<Tcp>`/`FlowPacket<Udp>` handlers
in `net_diagnostic` into one. Doesn't replace the typed
events — they're still useful when handlers genuinely need
the type-level guarantee.

### N3. `TcpRst` typed event

```rust
pub struct TcpRst {
    pub key: FlowKey,
    pub stats: FlowStats,
    pub ts: Timestamp,
}
```

Synthesised from `FlowEnded<Tcp>` when `reason == EndReason::Rst`.
The dispatch arm fires only the RST case. Same shape would
apply to a `TcpFinClean` event, an `IdleEvicted` event, etc. —
but RST is the operationally important one.

### N4. `IcmpError` typed event

```rust
pub struct IcmpError {
    pub kind: DestUnreachableKind,  // or a richer enum across all error types
    pub family: IcmpFamily,
    pub correlated_flow: Option<FlowKey>,  // from FlowTracker::lookup_inner
    pub ts: Timestamp,
}
```

Fires for any of Destination Unreachable / Time Exceeded /
Parameter Problem / Packet Too Big. Pre-classified, with the
flow-tracker join already done.

### N5. `Monitor::builder().all_l4()` umbrella

```rust
.all_l4()  // == .protocol::<Tcp>().protocol::<Udp>().protocol::<Icmp>()
.all_l7()  // adds Http + Dns + Tls + TlsHandshake (feature-gated)
```

Trivial sugar. Removes a foot-gun ("I registered Tcp and Udp
but forgot Icmp, why aren't my unreachables firing?").

### N6. `MonitorBuilder::bandwidth_by_app()` primitive

```rust
Monitor::builder()
    .interface(iface)
    .all_l4()
    .bandwidth_by_app()                      // registers the state slot
    .report(Duration::from_secs(5), |snap: &BandwidthSnapshot| {
        for (label, bytes_per_sec) in snap.top_n(10) {
            println!("{label}: {bytes_per_sec} B/s");
        }
    })
    .sink(StdoutSink::default())
    .run_until_signal().await?;
```

Internally: registers a `RollingRate<&'static str, u64>` state
slot, attaches `AnyFlowPacket` handlers, and emits a
`BandwidthSnapshot` event on the configured cadence.

This single primitive collapses ~80 LoC of the net_diagnostic
example.

### N7. `MonitorBuilder::report()` higher-level helper

```rust
pub fn report<R, F>(
    self,
    period: Duration,
    f: F,
) -> Self
where
    R: ReportSnapshot,
    F: Fn(&R) + Send + 'static,
```

Where `ReportSnapshot` is a derivable trait that pulls fields
out of registered state slots:

```rust
#[derive(ReportSnapshot)]
struct MyReport {
    #[from_state] bandwidth: AppBandwidth,
    #[from_state] resets: TcpResetCount,
    #[from_state] icmp: IcmpUnreachableCount,
}
```

Solves the "three state slots → ugly tick handler" pain (§1.7).
The derive auto-generates the borrow-by-Copy / borrow-by-ref
dance.

### N8. `Monitor::builder().on_icmp_error(handler)` sugar

```rust
.on_icmp_error(|err: &IcmpError, ctx: &mut Ctx<'_>| {
    if let Some(flow) = err.correlated_flow {
        ctx.emit_anomaly("FlowKilledByIcmp", Severity::Warning)
            .with("flow", &format!("{flow:?}"))
            .with("kind", err.kind.as_str())
            .emit();
    }
    Ok(())
})
```

Implicitly registers the `Icmp` protocol marker. Half the
example's ICMP code goes away.

### N9. `MonitorBuilder::on_tcp_reset(handler)` sugar

```rust
.on_tcp_reset(|rst: &TcpRst, ctx: &mut Ctx<'_>| { … })
```

Same story for the TCP RST signal.

### N10. `Tick` payload elision

Currently:

```rust
.tick(period, |_tick: &Tick, ctx: &mut Ctx<'_>| { … })
```

Add a `Handler<NoEvent>`-style variant so:

```rust
.tick(period, |ctx: &mut Ctx<'_>| { … })
```

works. The Handler trait already has `PayloadOnly`/`PayloadCtx`
markers — add a `CtxOnly` marker for the tick case.

## 5. Bigger redesign options

These break more than the small items above; useful only if
the maintainer wants a sharper 0.22 narrative than "polish".

### R1. Split `Protocol` into `FlowProtocol` + `MessageProtocol`

`Tcp`, `Udp` track flows (lifecycle + per-packet events).
`Http`, `Dns`, `Tls` extend flows (messages within a flow).
`Icmp` is neither — it's stateless message receipt.

```rust
pub trait FlowProtocol: Protocol { /* FlowStarted/Ended/Packet apply */ }
pub trait MessageProtocol: Protocol { /* only on::<Self> fires */ }
```

Then `.protocol::<Icmp>()` becomes `.listen_messages::<Icmp>()`
or similar. Fixes §1.12 architectural confusion. Migration
cost: low (the public `Protocol` trait stays; the new traits
are markers).

### R2. Drop the `FlowPacket<P>` parameterization

Replace with one type `FlowPacket { proto, key, side, len, tcp,
ts }`. Type-level dispatch on protocol was always vestigial for
per-packet events (you always end up branching on
`evt.key.proto` anyway). The parametric version stays for
lifecycle events where it earns its keep.

This goes further than N2's `AnyFlowPacket`: there is no other
shape. Pure simplification.

### R3. First-class "report" model alongside "anomaly"

The current architecture has two output streams:
- `AnomalySink` (event-driven, per-anomaly, structured)
- `EventStream<P>` (broadcast subscriber, per-message)

Reports are a third shape: periodic snapshot of derived state.
Today they squat on `Tick` + `println!`.

```rust
pub trait ReportSink {
    type Snapshot;
    fn write(&mut self, snap: &Self::Snapshot);
}
```

`StdoutReportSink<R>`, `JsonReportSink<R>`, `PrometheusReportSink<R>`,
`InfluxReportSink<R>`. Each `report()` registration declares
its `R` and the framework drives the cadence.

Suricata's `stats.log` is this shape. Zeek's `conn.log`
periodic summary is this shape. We've been forcing users to
roll it themselves.

### R4. Move bandwidth tracking off the user-handler hot path

A `bandwidth_by_app()` registration could install a *dispatcher
shim* that increments the right bucket **before** the user
handler chain runs, without taking a slot in
`Dispatcher::dispatch::<AnyFlowPacket>`. eBPF map-backed
storage (if running on a recent kernel) is the obvious
extension — Cilium/Hubble do this for K8s pod-level
bandwidth observability.

The user-facing API stays the same as N6; the implementation
is what changes. Pulls per-packet bookkeeping off the Rust hot
path entirely on supported kernels.

### R5. Generic `Histogram<K, V>` / `Bucketed<K, V>` primitives

`TimeBucketedCounter<K>` covers rate-based anomalies.
`RollingRate<K, V>` covers bandwidth-by-key.
`Bucketed<K, V>` would cover snapshots-by-key (top-N traffic
sources, top URL paths, top DNS domains). All three are
variants of "key → value over time" with different
read/write/decay characteristics.

Promoting them to a unified `correlate::` toolkit (instead of
the current bespoke `TimeBucketedCounter` + the proposed
`RollingRate` + the user's hand-rolled HashMap) collapses the
design space.

### R6. eBPF integration as a real path, not a sample

For the operational use cases `net_diagnostic` covers,
half the work could happen kernel-side:
- Bandwidth tracking via `BPF_MAP_TYPE_PERCPU_HASH` keyed by
  flow tuple, incremented by an XDP program.
- TCP RST detection at the kernel via `BPF_PROG_TYPE_SOCK_OPS`.
- ICMP error correlation via a kernel-side flow table lookup.

netring already has `xdp-loader`. The story would be:
"flowscope ships the BPF programs; netring loads them and
exposes the userland map readouts as if they were typed
events." Bigger lift, but the perf delta on a 10 Gbps NIC is
the difference between a working monitor and a dropped-packet
graveyard.

Cilium and Falco have shown this is feasible without
sacrificing the high-level user API.

## 6. Prior art notes (lightly researched)

What other tools do for these three signals:

**Suricata** ships an `IcmpEvent` JSON record for every parsed
ICMP message. Their flow log emits a `flow_state` field per
ended flow that includes "tcp_state" with values like
"closed", "reset", "fin_wait1". For bandwidth they have
`stats.log` (which we'd model as a `ReportSink`).

**Zeek** ships a `conn.log` with a per-flow `history` string
(`Sh` = SYN seen, `R` = RST, etc.) and a `conn_state` field
that already classifies the reset case. Their `icmp.log` is
separate. Stats roll up to `capture_loss.log` and
`stats.log`. The model maps near-1:1 to what we'd want as
"named report sinks".

**Wireshark** uses a "conversation" abstraction for
bandwidth-by-endpoint. Their `tshark -z conv,tcp` is the same
shape as the `BandwidthSnapshot` proposal — top-N talkers by
bytes.

**Cilium/Hubble** uses eBPF for L3/L4 observability inside
Kubernetes. They preprocess packets in-kernel and stream
processed flows to userland. Their flow record carries
"verdict" (FORWARDED / DROPPED / ERROR / AUDIT) which
includes ICMP error correlation — they explicitly join ICMP
back to the source flow. Worth borrowing the `Verdict` field
shape on a `FlowEnded` extension.

**Vector.dev** treats every observability concern as a
typed stream with sink adapters. The closest model to what
netring already has, but with a built-in "metrics" stream
type alongside "events". Their `internal_metrics` source is
the equivalent of our proposed `ReportSink`.

**nDPI** (community) and **Hyperscan** (Intel) classify
applications beyond port heuristics. flowscope already
defers nDPI-style DPI to user code; that's the right call,
but `protocol_label()` extensibility (§F7) would let users
plug nDPI without a flowscope fork.

## 7. Suggested 0.22 order

Tiers ordered by "ratio of (user value) to (LoC delta)":

**Tier 1 — sharp, immediate wins.** Maybe 2 days total.

- N1: prelude completeness
- N5: `all_l4()` / `all_l7()` umbrellas
- N9, N8: `on_tcp_reset()` / `on_icmp_error()` sugar (these
  depend on N3 + N4 being shipped)
- F2: surface `IcmpType::is_error()` in flowscope docs
- F4: `FlowStats` discoverability sweep
- F6: re-export hygiene

**Tier 2 — primitives. Maybe a week.**

- F1: `DestUnreachableKind`
- F3: `FiveTupleKey::app_label()`
- F8: `RollingRate<K, V>`
- N2: `AnyFlowPacket`
- N3: `TcpRst` event
- N4: `IcmpError` event (depends on F5)

**Tier 3 — Bandwidth-by-app shipped. Maybe 3 days.**

- N6: `bandwidth_by_app()` + `BandwidthSnapshot`
- N7: `report()` + `ReportSnapshot` derive
- R3: `ReportSink` trait (the minimal cut — stdout/JSON sinks
  only)

**Tier 4 — bigger redesigns.** Discretionary; pick one for
the 0.22 narrative.

- R1: split Protocol into FlowProtocol/MessageProtocol
- R2: drop FlowPacket<P> parameterization
- R5: unified `correlate::` primitives toolkit

**Tier 5 — research / spike. Time-boxed to a week.**

- R6: eBPF-backed bandwidth (XDP + percpu hash). Spike to
  validate the perf delta on a real NIC before committing to
  the redesign.
- F5: `FlowTracker::lookup_inner` for ICMP correlation
  (requires a flowscope-side index expansion)

Tier 1+2+3 takes the example back from 306 LoC to roughly
70–80 LoC — the headline win. Tier 4 would take it under 50.
Tier 5 changes the underlying performance ceiling.

## 8. Open questions

For the maintainer (you):

1. **Should ICMP keep its `Protocol` registration ceremony?**
   If we ship `MessageProtocol` (R1), ICMP migrates and the
   ceremony reads differently. If we don't, we should at
   least add a doc warning that ICMP fires `on::<Icmp>` and
   does **not** fire FlowStarted/FlowEnded.

2. **Is `bandwidth_by_app` the only "report" use case worth a
   builtin, or do we want a generic `report()` first?**
   Building `bandwidth_by_app` on top of a generic `report()`
   primitive (N7 + N6) is cleaner. Building the bandwidth
   case first and generalizing later means the generic API
   gets shaped by one concrete use case — risky.

3. **eBPF: how aggressive?**
   The `xdp-loader` feature exists but is minimally wired.
   Going from "loads a redirect-all XDP program" to "loads
   per-feature BPF programs that feed userland maps" is a
   strategy shift, not an increment. Worth a Phase of its own.

4. **Custom port labels: who owns the table?**
   F7 needs an answer: does flowscope ship a registry?
   Does netring carry per-deployment overlays? Does each
   `Monitor::builder()` own its table? My instinct is "ship
   `LabelTable` as a flowscope type, pass it through the
   netring builder" — but worth thinking through.

5. **Are we OK shipping 0.22 with 8+ new event types in the
   prelude?**
   `FlowPacket`, `FlowTick`, `ParserClosed`, `AnyFlowPacket`,
   `AnyFlowEnded`, `TcpRst`, `IcmpError`, `BandwidthSnapshot`,
   plus existing types. The prelude grows. Acceptable, or
   refactor into sub-preludes (`netring::prelude::monitor`,
   `netring::prelude::icmp`)?

## 9. Non-recommendations

For completeness, things I considered and decided against:

- **YAML/Lua config DSL.** Already declined in 0.21
  retrospective. Rust handlers compose better than any DSL
  for this domain.
- **Builtin Prometheus exporter.** `MetricsSink` already
  bridges to the `metrics` facade. Exporters are downstream
  user code.
- **HTTP/2 + QUIC parsers.** Strategic deferral inherited from
  flowscope.
- **Cross-shard state merging as a primitive.** Already in
  `plans/netring-0.22-roadmap.md` §C.5 — separate work.
