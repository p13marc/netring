## §2 Plan 160 — `KeyIndexed::drain_expired(now)` iterator

**Priority:** P0. Already on the netring 0.22 roadmap as
`0.22-G` — netring's local `correlate::KeyIndexed` is held
**solely** because this method doesn't exist upstream yet.
Shipping it lets netring drop its local copy and re-export.

**Source line:** `flowscope/src/correlate/indexed.rs:22` —
`pub struct KeyIndexed<K, V>`.

**Proposed shape:**

```rust
impl<K, V> KeyIndexed<K, V>
where
    K: Hash + Eq,
{
    /// Drain entries whose TTL has elapsed at `now`. Returns
    /// an iterator yielding owned `(K, V)` pairs in arbitrary
    /// order. Entries that are not yet expired stay in the
    /// index. Companion to the existing `evict_expired(now)`
    /// (which discards) — `drain_expired` lets the caller
    /// inspect expired entries (commonly for the "DNS resolved
    /// but no connection followed" / "ICMP didn't explain
    /// a flow death" patterns).
    pub fn drain_expired(
        &mut self,
        now: Timestamp,
    ) -> impl Iterator<Item = (K, V)> + '_;
}
```

**Why ownership:** netring's `KeyIndexed` consumers
(`dns_resolved_no_connection`, `slow_tls_handshake`, the
hypothetical `IcmpUnexplainedDrop` correlator) need to
**own** the expired key/value so they can `emit` an anomaly
carrying it. Borrowing forces a copy at the call site anyway.

**Why an iterator (not a `Vec`):** zero-alloc on the happy
path (no expired entries → empty iterator). The dhat bench
in netring would notice any allocation here.

**Tests:** existing `TimeBucketedSet::evict_expired` and
`KeyIndexed::evict_expired` tests are the structural
template.

## §3 Plan 161 — `FlowTracker::lookup_inner(IcmpInner)` ICMP-to-flow join

**Priority:** P0. Highest-leverage addition in the entire
wishlist. Currently flowscope ships `IcmpType::error_inner()
-> Option<(&'static str, &IcmpInner)>` (returning the
embedded original 5-tuple) but no way to **join** that back
to the live flow table. Every L4 monitor wants this; today
each consumer rolls a hand-built `HashMap<FlowKey,
FlowStats>` cache mirroring flowscope's internal index.

**Source line:** `flowscope/src/tracker.rs` (whole file —
`FlowTracker` is the primary join point).
`flowscope/src/icmp/types.rs:276` — `IcmpInner { src, dst,
proto, src_port, dst_port }`.

**Proposed shape:**

```rust
impl<K> FlowTracker<K>
where
    K: FlowKey + Clone,
{
    /// Look up a live flow that matches the partial 5-tuple
    /// embedded in an `IcmpInner`. Returns the canonical
    /// `K` if the flow exists in the tracker, or `None` if
    /// no flow matches (truncated embed, parsing error, or
    /// the flow already expired).
    ///
    /// The lookup is direction-agnostic: an ICMP error
    /// reporting `(srcA, srcP) → (dstA, dstP)` matches a
    /// flow regardless of which direction the flow's
    /// canonical 5-tuple records.
    ///
    /// O(1) with a single hash lookup. Read-only — does not
    /// mutate the tracker.
    pub fn lookup_inner(&self, inner: &IcmpInner) -> Option<K>;

    /// Companion: read the current `FlowStats` for a
    /// matching flow, if any. Saves the second lookup for
    /// the common "join then read stats" pattern.
    pub fn stats_for_inner(
        &self,
        inner: &IcmpInner,
    ) -> Option<(K, &FlowStats)>;
}
```

**Why direction-agnostic:** ICMP errors carry the **original**
packet's `(src, dst)` ordering. A live flow's canonical
ordering (initiator vs responder) depends on which side sent
the first packet. Without direction-agnostic lookup, the
caller has to try both orderings — duplicating the index's
internal canonicalisation logic.

**Implementation cost on flowscope side:** small. The tracker
already keys flows by canonical 5-tuple. The new method
canonicalises the `IcmpInner` tuple (or tries both
orderings) and probes the existing index. No new index
structure needed.

**Unblocks (netring side):**
- `netring::protocol::event_typed::IcmpError { correlated_flow:
  Option<FlowKey>, ... }` typed event
- `Monitor::builder().on_icmp_error(handler)` sugar
- Reframes `examples/anomaly/icmp_explained_drop.rs` (which
  hand-rolls this join today)

**Caveat:** if the flow tracker doesn't currently expose a
public read-only lookup, this may require a small refactor
to split read-from-write. The existing public API tends to be
mutate-only (`track`, `flush`, etc.) — confirm before
estimating.

## §4 Plan 162 — `DestUnreachableKind` enum + accessor

**Priority:** P0. Cheap, mechanical, eliminates the
30-line v4/v6 classifier I wrote in `net_diagnostic.rs:108-140`
and any consumer will write the same way.

**Source lines:**
- `flowscope/src/icmp/types.rs:118` — `Icmpv4DestUnreachCode` (17 variants)
- `flowscope/src/icmp/types.rs:225` — `Icmpv6DestUnreachCode` (8 variants)

**Proposed shape:**

```rust
// flowscope::icmp::DestUnreachableKind
#[derive(Debug, Clone, Copy, PartialEq, Eq)]
#[cfg_attr(feature = "serde", derive(serde::Serialize, serde::Deserialize))]
#[non_exhaustive]
pub enum DestUnreachableKind {
    /// v4 `Host`/`DestHostUnknown`, v6 `AddressUnreachable`
    Host,
    /// v4 `Port`, v6 `PortUnreachable`
    Port,
    /// v4 `Net`/`DestNetworkUnknown`, v6 `NoRoute`
    Network,
    /// v4 `Protocol` — no v6 equivalent
    Protocol,
    /// v4 `*Prohibited`/`CommunicationProhibited`,
    /// v6 `AdminProhibited`/`RejectRouteToDestination`/`SourceAddressFailedIngressPolicy`
    AdministrativelyProhibited,
    /// v4 `FragmentationNeeded { mtu }` (mtu lost in the
    /// canonical mapping; user code that needs the MTU
    /// should match on `Icmpv4Type` directly).
    /// v6 has no exact equivalent — `PacketTooBig` is a
    /// separate type (2), not a code under DU.
    FragmentationNeeded,
    /// Anything else (Source Route Failed, NetworkTos, etc.)
    Other,
}

impl IcmpType {
    /// Classify a Destination Unreachable into a unified
    /// v4/v6 vocabulary. Returns `None` for non-DU types
    /// (Echo, TimeExceeded, ParameterProblem, etc.).
    pub fn dest_unreachable_kind(&self) -> Option<DestUnreachableKind>;
}

impl DestUnreachableKind {
    /// Short slug for metric labels.
    pub fn as_str(&self) -> &'static str {
        match self {
            Self::Host => "host_unreachable",
            Self::Port => "port_unreachable",
            Self::Network => "network_unreachable",
            Self::Protocol => "protocol_unreachable",
            Self::AdministrativelyProhibited => "administratively_prohibited",
            Self::FragmentationNeeded => "fragmentation_needed",
            Self::Other => "dest_unreachable_other",
        }
    }
}
```

**Decision required:** does `PacketTooBig` (ICMPv6 type 2)
fold into `FragmentationNeeded`, or stay separate?
Operationally they're the same signal ("MTU mismatch on path").
My recommendation: ship `DestUnreachableKind` first, then add
a parallel `IcmpType::mtu_signal() -> Option<MtuSignalKind>`
in 0.15 if a use case asks for it.

**Open question:** prelude inclusion. The new enum is the
sort of thing a netring `on_icmp_error` handler imports
constantly. Worth `pub use icmp::DestUnreachableKind` at
the crate root, or stay nested?

## §5 Plan 163 — `FiveTupleKey::app_label()` (always-Some)

**Priority:** P1. Removes the one ergonomic drag in
`net_diagnostic.rs:142-157` — the `is_tcp: bool` parameter
on `bandwidth_bucket` exists only because `protocol_label()`
is `Option`-returning and the L4 fallback has to happen at
the call site.

**Source line:** `flowscope/src/extract/five_tuple.rs:91` —
`pub fn protocol_label(&self) -> Option<&'static str>`.

**Proposed shape:**

```rust
impl FiveTupleKey {
    /// Always-some companion to [`Self::protocol_label`].
    /// Falls back to the canonical L4 name when neither
    /// port matches the well-known table:
    /// - `L4Proto::Tcp` → `"tcp"`
    /// - `L4Proto::Udp` → `"udp"`
    /// - `L4Proto::Icmp` / `L4Proto::IcmpV6` → `"icmp"`
    /// - `L4Proto::Sctp` → `"sctp"`
    /// - `L4Proto::Other(_)` → `"other"`
    ///
    /// Use this for bandwidth-by-app / metric labels where
    /// "we don't know the app, but we know the L4" is the
    /// right fallback. Use `protocol_label()` when an L7
    /// label is the *only* acceptable answer.
    pub fn app_label(&self) -> &'static str {
        self.protocol_label()
            .unwrap_or_else(|| self.proto.canonical_name())
    }
}

impl L4Proto {
    /// Stable short slug for any `L4Proto`. Exposed so
    /// `app_label`-style call sites in user code can reuse
    /// the canonical names without round-tripping through
    /// `FiveTupleKey`.
    pub fn canonical_name(&self) -> &'static str;
}
```

**Why expose `L4Proto::canonical_name()` separately:** it's
a primitive the test helpers use too. Externalising it
prevents users from rolling their own `match L4Proto { … }`
in 4 places.

## §6 Plan 164 — `correlate::RollingRate<K, V>` primitive

**Priority:** P1. The 0.22 design input flags bandwidth-by-app
as the operationally-most-common monitor pattern. flowscope
already ships `TimeBucketedCounter<K>` (rate-based anomalies)
and `Ewma<K>` (exponential moving average) — what's
missing is the **rolling rate** primitive that combines
"key + value-sample + window" into a snapshot iterator.

**Source line:** `flowscope/src/correlate/bucketed.rs:23` —
`pub struct TimeBucketedCounter<K>` (closest existing
shape; adds 1 per bump rather than a user-supplied value).

**Proposed shape:**

```rust
// flowscope::correlate::RollingRate
pub struct RollingRate<K, V>
where
    K: Hash + Eq,
    V: Default + Copy + std::ops::AddAssign + From<u64> + Into<f64>,
{
    window: Duration,
    bucket_width: Duration,
    // For each K, a ring of buckets holding accumulated V.
    by_key: HashMap<K, RateBuckets<V>>,
}

impl<K, V> RollingRate<K, V>
where
    K: Hash + Eq + Clone,
    V: Default + Copy + std::ops::AddAssign + From<u64> + Into<f64>,
{
    /// Window size 60s, bucket width 1s (matches the
    /// Prometheus rate(...) default).
    pub fn new_unbounded(window: Duration, bucket_width: Duration) -> Self;

    /// Add `v` to the current bucket for `k` at time `now`.
    /// Old buckets outside the window are lazily zeroed on
    /// the next access.
    pub fn record(&mut self, k: K, v: V, now: Timestamp);

    /// `sum(V over last window) / window_secs` — per-second
    /// rate. Returns `0.0` if `k` has no samples in the
    /// window.
    pub fn rate(&self, k: &K, now: Timestamp) -> f64;

    /// Snapshot all keys with non-zero rate. Iterator
    /// yields `(K, rate_per_sec)` pairs in arbitrary order.
    /// Useful for "top-N talkers" reports — pair with the
    /// existing `correlate::TopK<K>` to get the actual top-N.
    pub fn snapshot(
        &self,
        now: Timestamp,
    ) -> impl Iterator<Item = (&K, f64)> + '_;

    /// Drop buckets outside the window. Idempotent.
    /// Mirrors `TimeBucketedCounter::evict_expired`.
    pub fn evict_expired(&mut self, now: Timestamp);
}
```

**Design notes:**

- **Why generic `V`:** the same primitive serves both
  bandwidth (`V = u64` for bytes/sec) and request-rate
  (`V = u64` for count/sec) and even latency-sum
  (`V = u64` for total-ns/sec, divided externally for
  mean-latency).
- **Why `From<u64>` + `Into<f64>`:** the bound's only
  purpose is the `rate()` arithmetic. Restricting to numeric
  types is fine; the user can wrap a newtype if they want
  semantic separation.
- **Capacity vs unbounded:** ship `new_unbounded` first
  (parallel to `KeyIndexed::new_unbounded`); add a
  `with_capacity(_, _, n)` constructor only when a user hits
  the LRU need.
- **Allocation contract:** netring's dhat bench will
  catch any per-bump allocation. The implementation should
  reuse bucket arrays — same trick `TimeBucketedCounter`
  already uses.

**Composition example:**

```rust
// Inside a netring monitor handler:
let bw = ctx.state_mut::<RollingRate<&'static str, u64>>();
bw.record(evt.key.app_label(), evt.len as u64, ctx.ts);

// In the tick handler:
for (label, bytes_per_sec) in bw.snapshot(ctx.ts) {
    println!("{label}: {bytes_per_sec:.0} B/s");
}
```

**Naming alternative:** `RollingSum<K, V>` if `Rate` reads
too Prometheus-flavoured. The "rate" framing is closer to
what users will reach for; pick one and stay consistent
with the existing `TimeBucketedCounter`'s vocabulary.

## §7 Plan 165 — `protocol_label` extensibility

**Priority:** P1. The `well_known/mod.rs` table is great for
the standard ports but has no escape hatch for site-custom
services. Every real deployment has at least one
("our internal gRPC service on 8765", "our metrics scrape
on 9101", "the legacy app on 30443"). Today the only options
are: fork the table, or shadow `protocol_label()` with a
user wrapper that re-implements the dispatch logic.

**Source line:** `flowscope/src/well_known/mod.rs:53` —
`pub fn protocol_label(proto, src_port, dst_port) -> Option<&'static str>`.

**Proposed shape — Option A (preferred): label table type**

```rust
// flowscope::well_known
#[derive(Clone, Default)]
pub struct LabelTable {
    overrides: HashMap<(L4Proto, u16), &'static str>,
    /// If `true` (default), unknown ports fall back to the
    /// built-in table. If `false`, only `overrides` are
    /// consulted.
    inherit_builtin: bool,
}

impl LabelTable {
    pub fn new() -> Self;
    pub fn standalone() -> Self; // inherit_builtin = false

    /// Add/override a label. The `&'static str` constraint
    /// keeps the zero-allocation contract.
    pub fn set(&mut self, proto: L4Proto, port: u16, label: &'static str) -> &mut Self;

    /// Bulk-set from an iterator — useful for config-driven
    /// table population.
    pub fn extend<I>(&mut self, entries: I) -> &mut Self
    where
        I: IntoIterator<Item = (L4Proto, u16, &'static str)>;

    /// Lookup. Same shape as the existing free function.
    pub fn lookup(&self, proto: L4Proto, src_port: u16, dst_port: u16)
        -> Option<&'static str>;
}

impl FiveTupleKey {
    /// Companion to `protocol_label()` that consults a
    /// caller-provided table first.
    pub fn protocol_label_with(&self, table: &LabelTable) -> Option<&'static str>;

    /// And the always-Some variant:
    pub fn app_label_with(&self, table: &LabelTable) -> &'static str;
}
```

**Why not a closure-based `protocol_label_with(custom: &impl Fn(...))`:**
the function-table shape composes better than a closure for
typical use cases (load from config, mutate at runtime).
Plus a `LabelTable` value is `Clone`able and `Send` — a
closure usually isn't.

**Optional 0.15 extension:** `FlowTracker::with_label_table(table)`
so the label propagates through the tracker — but I'd ship
the `LabelTable` value type first and see whether the
tracker integration is actually needed before adding it.

## §8 Plan 166 — `flowscope::icmp::types` re-export hygiene

**Priority:** P2 (low — pure docs nuisance). Worth mentioning
because the 0.21 sweep had to chase one of these.

**Source line:** `flowscope/src/icmp/mod.rs:73` —
`mod types;` followed by `pub use types::*;` at line 77.

**Symptom:** the rustdoc-generated path
`flowscope::icmp::types::Icmpv6DestUnreachCode` is
**reachable in autocompletion** (because IDEs index it) but
**broken in code** (the `types` module is private; only
`pub use types::*` rescues access via the parent path). When
I first hit it I wrote `use flowscope::icmp::types::Icmpv6DestUnreachCode`
and got a compile error pointing at a private module.

**Proposed fix (pick one):**

- **A.** Promote `types` to `pub mod types`. The `pub use
  types::*` shim stays for backward compat. Both paths work,
  rustdoc stops lying.
- **B.** Keep `types` private but add `#[doc(hidden)]` or
  similar so the rendered docs only show the canonical
  `flowscope::icmp::*` path.

Option A is more permissive; option B is tighter. Either
fixes the bait-and-switch. No netring code changes needed
post-fix.

**Time cost:** 30 minutes including the test-pass.

## §10 Plan 168 — `FlowSide`-aware byte split (optional)

**Priority:** P3 (nice-to-have). The `net_diagnostic`
example dropped `evt.side` because the bandwidth-by-app
case doesn't care which side a packet came from. But "what's
the directional throughput split for this flow?" is a real
monitor question, and the current `FlowStats` exposes
`bytes_initiator`/`bytes_responder` as fields.

**Proposed shape:**

```rust
impl FlowStats {
    /// Bytes attributed to the given side.
    pub fn bytes_for(&self, side: FlowSide) -> u64 {
        match side {
            FlowSide::Initiator => self.bytes_initiator,
            FlowSide::Responder => self.bytes_responder,
        }
    }

    /// Packets attributed to the given side.
    pub fn pkts_for(&self, side: FlowSide) -> u64;

    /// Mean packet size for the given side. Returns `0.0`
    /// if the side has zero packets.
    pub fn mean_pkt_size_for(&self, side: FlowSide) -> f64;

    /// Direction skew: `(bytes_initiator - bytes_responder) / total_bytes`.
    /// Returns `0.0` for empty flows. Positive = initiator
    /// heavy (uploads), negative = responder heavy
    /// (downloads). Useful for detecting one-sided flows.
    pub fn direction_skew(&self) -> f64;
}
```

Pure sugar over existing fields; no perf concern, no
breaking change. Drop if the maintainer thinks the field
access is already terse enough.

## §12 Items explicitly **not** in this wishlist

For clarity on what netring 0.22 will handle netring-side
without asking flowscope:

- **R1 (split `Protocol` → `FlowProtocol` + `MessageProtocol`)** —
  pure netring-side trait split. flowscope's `Driver<E>` shape
  already supports both modes; the netring marker trait is
  the only change.
- **R2 (drop `FlowPacket<P>` parameterization)** — netring-side.
  flowscope's per-packet hook signature stays as-is.
- **R3 (`ReportSink` trait)** — netring-side. flowscope is
  runtime-free; report cadence + sink lifecycle belong in
  netring.
- **R4 (eBPF-backed bandwidth)** — out of scope for 0.14.

  driven by a dedicated `flowscope-ebpf` companion crate
  rather than the core. Cilium/Hubble pattern as prior art.
- **JA4 family / IPFIX / HTTP/2 / QUIC** — flowscope's
  strategic deferrals; nothing in the netring 0.22 work asks
  for them.

## §13 Open questions for the maintainer

1. **Plan 161 (`lookup_inner`) — does the current
   `FlowTracker` expose a public read-only API?** If
   everything is mutate-only (`track`, `flush`, etc.) the
   refactor cost grows. Quick spike to confirm before
   estimating the cycle.

2. **Plan 162 (`DestUnreachableKind`) — `FragmentationNeeded`
   vs `PacketTooBig` collation?** v6 `PacketTooBig` is type
   2, not a code under DU — so the v4-style "DU code = needs-frag"
   doesn't map cleanly. Two reasonable answers:
   - Map both into `FragmentationNeeded` (operationally
     identical signal)
   - Ship `DestUnreachableKind` first, add a parallel
     `IcmpType::mtu_signal()` in 0.15 if asked
   My pick is the second — keeps the kind enum's scope
   tight to actual DU codes.

3. **Plan 164 (`RollingRate`) — `RollingRate` vs `RollingSum`
   naming?** The Prometheus-flavoured "rate" framing is
   familiar but implies a `per_second` denominator; the
   underlying primitive is really a "sum-over-window". My
   pick: ship as `RollingRate` since `rate()` is the
   primary accessor users will hit; if the bare sum is more
   useful (e.g. "bytes-in-this-minute" without the per-sec
   divide), add a `RollingRate::sum(k, now) -> V` companion.

4. **Plan 165 (`LabelTable`) — should it be `Send + Sync`?**
   If yes, `&'static str` labels suffice. If users want
   runtime-loaded labels from a config file, we'd need
   `Cow<'static, str>` and the table becomes !Sync via
   interior mutability. Stay `&'static str` for 0.14 (matches
   the existing well_known table's contract); revisit if
   demanded.

5. **Plan 167 (discoverability) — `pub use` everything to
   the crate root, or stay namespaced?** netring's prelude
   is already 30+ names; flowscope's prelude could match if
   the maintainer is OK with the noise. Alternative: a
   `flowscope::prelude::reports` sub-prelude for the
   stats/correlate primitives that examples reach for.
